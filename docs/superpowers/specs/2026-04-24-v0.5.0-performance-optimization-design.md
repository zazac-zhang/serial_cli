# v0.5.0 Performance Optimization Design

**Date**: 2026-04-24
**Status**: In Progress
**Version**: v0.5.0-dev

## Overview

This document describes the performance optimization work for Serial CLI v0.5.0, focusing on benchmark-driven optimization to identify and address actual performance bottlenecks.

---

## Goals

1. **Establish baseline performance metrics** - Create comprehensive benchmarks for all major operations
2. **Identify bottlenecks** - Use real-world scenarios to find actual performance issues
3. **Optimize critical paths** - Focus on data transfer, startup time, and memory usage
4. **Maintain API compatibility** - Conservative approach with no breaking changes
5. **Enable continuous monitoring** - Make it easy to detect performance regressions

---

## Architecture

### Benchmark Module Structure

```
src/benchmark/
├── mod.rs          # Public exports and category definitions
├── runner.rs       # Benchmark execution engine
└── reporter.rs     # Result reporting and comparison
```

**Key Components**:

- **BenchmarkRunner**: Executes benchmarks with configurable iterations
- **BenchmarkResult**: Stores timing and throughput data
- **BenchmarkReport**: Aggregates results with timestamp
- **ComparisonResult**: Compares two runs with regression detection

### CLI Integration

New `benchmark` command with subcommands:
- `run <category>` - Execute benchmarks (serial-io, virtual-port, protocol, startup, memory, concurrency, all)
- `compare <baseline> <current>` - Compare results and detect regressions
- `list` - Show available benchmark categories

## Benchmark Categories

### 1. Serial I/O Benchmarks

**Purpose**: Measure data transfer throughput and latency

**Benchmarks**:
- Buffer copy throughput (64B - 16KB)
- Serial port write latency
- Serial port read latency
- Round-trip echo latency
- Continuous streaming throughput

**Metrics**: MB/s, ns/operation

### 2. Virtual Port Benchmarks

**Purpose**: Measure virtual port creation and bridging performance

**Benchmarks**:
- Virtual port creation time
- Bridge throughput (data transfer between port pairs)
- AsyncFd polling efficiency
- Health check overhead

**Metrics**: ns/iter, MB/s, events/sec

### 3. Protocol Benchmarks

**Purpose**: Measure protocol parsing and processing performance

**Benchmarks**:
- Modbus RTU frame parsing
- Modbus ASCII frame parsing
- AT Command parsing
- Line protocol parsing
- Custom Lua protocol execution

**Metrics**: frames/sec, ns/frame, MB/s

### 4. Startup Benchmarks

**Purpose**: Measure application initialization time

**Benchmarks**:
- Cold start time (no config)
- Warm start time (with config)
- Protocol loading time
- Lua engine initialization

**Metrics**: ms to startup

### 5. Memory Benchmarks

**Purpose**: Measure memory allocation patterns and peak usage

**Benchmarks**:
- Peak memory during startup
- Memory per virtual port pair
- Memory per active protocol
- Buffer allocation rate
- Memory leak detection (long-running)

**Metrics**: MB, allocations/sec

### 6. Concurrency Benchmarks

**Purpose**: Measure performance under concurrent operations

**Benchmarks**:
- Multiple virtual port pairs
- Concurrent protocol parsing
- Parallel Lua script execution
- Thread contention analysis

**Metrics**: ops/sec, lock contention %

## Optimization Strategy

### Phase 1: Baseline Establishment ✅ Complete

- Create benchmark infrastructure
- Implement basic benchmarks for each category
- Establish baseline metrics
- Implement result persistence and comparison

### Phase 2: Bottleneck Identification 🚧 In Progress

- Run benchmarks on representative hardware
- Profile hot paths with flamegraphs
- Identify top 3 bottlenecks per category
- Prioritize optimization targets

### Phase 3: Targeted Optimization ⏳ Pending

**Data Transfer Optimizations**:
- Zero-copy data transfer using `bytes::Bytes`
- AsyncFd polling optimization (reduce syscalls)
- Buffer size tuning based on measurements
- Batch read/write operations

**Startup Optimizations**:
- Lazy initialization of non-essential components
- Parallel loading of protocols
- Defer Lua engine setup until first use
- Cache compiled bytecode

**Memory Optimizations**:
- Buffer reuse with memory pools
- Reduce allocations in hot paths
- Use stack-allocated buffers where possible
- Implement custom allocator for buffers

### Phase 4: Validation ⏳ Pending

- Re-run all benchmarks
- Verify improvements meet targets
- Check for regressions in other areas
- Update baseline metrics

## Performance Targets

### Data Transfer

- **Buffer copy**: >30 GB/s for 16KB buffers (currently ~38 GB/s ✅)
- **Serial I/O**: <100ns overhead per operation
- **Virtual bridge**: <10µs latency per transfer

### Startup Time

- **Cold start**: <50ms
- **Warm start**: <20ms
- **Protocol load**: <5ms per protocol

### Memory Usage

- **Base footprint**: <10 MB
- **Per virtual port**: <1 MB
- **Per protocol**: <500 KB

## Usage Examples

### Running Benchmarks

```bash
# List available categories
serial-cli benchmark list

# Run specific category with custom iterations
serial-cli benchmark run serial-io --iterations 1000

# Run all benchmarks and save results
serial-cli benchmark run all --output baseline.txt

# Compare results to detect regressions
serial-cli benchmark compare baseline.txt current.txt
```

## Regression Detection

The `benchmark compare` command automatically detects performance regressions:

- **Regression threshold**: 5% slower than baseline
- **Improvement threshold**: 5% faster than baseline
- **Output**: Shows regressions (❌), improvements (✅), and unchanged (→)

Example:
```
=== Benchmark Comparison ===

  buffer_copy_16384: ❌ REGRESSION (+8.3%)
    Baseline: 38098.64 ns/iter → Current: 41234.12 ns/iter

Summary:
  Regressions: 1
  Improvements: 0
  Unchanged: 6
```

## Design Decisions

### Why Criterion.rs for CLI Benchmarks?

We chose to build a custom benchmark runner instead of using criterion.rs directly for the CLI because:

1. **Simplicity**: Custom runner is easier to integrate with CLI
2. **Control**: Full control over output format and persistence
3. **Flexibility**: Easy to add custom metrics (throughput, memory)
4. **Criterion still available**: The existing `benches/` directory still uses criterion.rs for detailed profiling

### Why Conservative Optimization?

1. **Safety**: Performance optimizations often introduce bugs
2. **Compatibility**: No breaking changes for users
3. **Measurable**: Each optimization should show measurable improvement
4. **Reversible**: Easy to revert if issues arise

### Why Focus on Data Transfer First?

1. **Core functionality**: Data transfer is the primary use case
2. **User-visible**: Direct impact on user experience
3. **Measurable**: Easy to benchmark and compare
4. **High impact**: Small improvements yield big gains at scale
